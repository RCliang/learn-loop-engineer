任务二：给 Agent 装上手脚，让它能操作文件和命令

// 给 Agent 装上手 —— 三件事

① 告诉 LLM 它有什么工具（Tools Schema）

② 解析 LLM 返回的工具调用请求

③ 执行工具，把结果回传给 LLM

LLM 思考
→
调用工具
→
执行
→
回传结果
→
循环
🧙
任务二续：定义工具的样子，LLM 才知道怎么调

// 工具定义：告诉 LLM 它能做什么

TOOLS = [{ "type": "function", "function": { "name": "bash_exec", "description": "在工作目录中执行 shell 命令", "parameters": { "properties": { "command": {"type": "string"} } } } }] # description 写得越清晰 # → LLM 选对工具的概率越高
🧙
任务二续：怎么判断"做完了"？不调工具就是完成

// 关键判断：何时算"完成"

# 当 LLM 不再调用任何工具 # → 它认为任务已经完成了 if not msg.tool_calls: print("✓ 任务完成") return text # 这是 function calling 的隐式约定： # "不调工具" = "我做完了"

任务三续：三道防线，从暴力到智能

// 三层刹车

class Evaluator: def should_stop(self, turn, last_action): # 刹车 1：硬上限（保险丝） if turn >= self.max_turns: return True, "max_turns" # 刹车 2：死循环检测（同一操作 ≥3 次） h = md5(str(last_action)) if self._hashes.count(h) >= 2: return True, "loop_detected" # 刹车 3：self-critique（下一课展开） return False, ""

Agent 可靠性的关键 ≠ LLM 更强，= Evaluator 更靠谱

🧙
任务四：格式化结果，让 Agent 看懂工具的反馈

// 格式化：让 LLM 看得懂结果

原始结果（LLM 需要费力解析）：

{"ok":false,"error_type":"TimeoutExpired","message":"超时 30s","stdout":"","stderr":"","exit_code":-1,"duration_s":30.012}

格式化后（一目了然）：

[错误] bash_exec 失败: 命令超时(30s)
def format_observation(tool_name, result): if not result.get("ok"): return f"[错误] {tool_name} 失败: {result['message']}" if tool_name == "bash_exec": return f"退出代码: {result['exit_code']}\n输出:\n{result['stdout']}" # 目标不是美观，是让 LLM 做出更好的下一步决策
